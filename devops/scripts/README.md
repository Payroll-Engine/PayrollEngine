# DevOps Scripts

PowerShell scripts for PayrollEngine release preparation and database schema management.

## Directory Layout

These scripts expect the standard PayrollEngine repository layout:

```
<ReposRoot>\
  PayrollEngine\              ← this repo (with devops\scripts\)
  PayrollEngine.Backend\
  PayrollEngine.Core\
  PayrollEngine.Client.Core\
  PayrollEngine.Client.Scripting\
  PayrollEngine.Client.Services\
  ...
```

By default, scripts resolve `ReposRoot` three levels up from the script location
(`devops\scripts\` → repo root → parent). Override with `-ReposRoot` if your layout differs.

## Global Usage

To run scripts from any directory without specifying the full path, add the scripts folder
to `PATH` in your PowerShell profile.

Run once in a PowerShell session to create the profile file and add the scripts folder (adjust the path to match your layout):

```powershell
New-Item -ItemType File -Path $PROFILE -Force
Add-Content -Path $PROFILE -Value "`$env:PATH += `";C:\Shared\PayrollEngine\Repos\PayrollEngine\devops\scripts`""
```

After restarting the session the scripts are available from any directory — without path prefix:

```powershell
Compose-DbScript.ps1
Update-BreakingChanges.ps1
```

> Note: Do not use `.\` — that forces PowerShell to look in the current directory only.
> Parameters use a single dash: `-ScriptsDir`, not `--ScriptsDir`.

## Release

Scripts for release preparation, changelog consolidation and breaking change detection.

| Script | README | Purpose |
|--------|--------|---------|
| `Update-BreakingChanges.ps1` | [README](Update-BreakingChanges.README.md) | Detect breaking API changes and document them in `RELEASE_NOTES.md` |
| `Release-Changelog.ps1` | [README](Release-Changelog.README.md) | Consolidate Git changelogs across all repositories for a release |
| `Update-CommitMessage.ps1` | [README](Update-CommitMessage.README.md) | Populate the VS commit message from the latest `RELEASE_NOTES.md` section |

### Update-BreakingChanges

**Update-BreakingChanges.ps1** detects breaking changes in uncommitted code across all
PayrollEngine APIs, then documents them in `RELEASE_NOTES.md`. It regenerates `swagger.json`
from the current Backend build via Swashbuckle CLI, compares the REST API spec and .NET
assembly surfaces against the last Git commit, and appends findings to the release notes.

Checks performed:
- Swagger generation from `PayrollEngine.Backend.Server`
- REST API (Provider) — `swagger.json` diff (endpoints, parameters, schemas, response codes)
- Backend API Models — public .NET API surface diff via `MetadataLoadContext`
- Scripting API (Regulator) — public .NET API surface diff
- Client Services (Automator) — public .NET API surface diff
- `RELEASE_NOTES.md` — append detected breaking changes

Breaking changes are automatically appended to `RELEASE_NOTES.md`, mapped to the correct
section (Backend, Libraries/Client Scripting, Libraries/Client Services). Only new entries
are added; existing content is never modified.

```powershell
# against HEAD (default), auto-updates RELEASE_NOTES.md
.\Update-BreakingChanges.ps1

# against a specific tag
.\Update-BreakingChanges.ps1 -BaselineRef "v0.9.0"

# with confirmation prompt before updating RELEASE_NOTES.md
.\Update-BreakingChanges.ps1 -ConfirmReleaseNotes

# skip RELEASE_NOTES.md update entirely
.\Update-BreakingChanges.ps1 -SkipReleaseNotes

# with Markdown report
.\Update-BreakingChanges.ps1 -ReportPath ".\breaking-changes.md"

# use pre-generated swagger.json (skip Swashbuckle CLI generation)
.\Update-BreakingChanges.ps1 -SwaggerJsonPath "docs\swagger.json"
```

The `-SwaggerJsonPath` parameter accepts a path to a pre-generated `swagger.json` file.
When provided, the script skips the Swashbuckle CLI build step and uses the file directly
for the REST API diff. Useful in CI environments where `swagger.json` has already been
generated by a prior build step (e.g. Wave 5 in the release pipeline).

Prerequisites: .NET SDK, Swashbuckle CLI (`dotnet tool install -g Swashbuckle.AspNetCore.Cli`).

### Release-Changelog

**Release-Changelog.ps1** consolidates Git changelogs across all repositories for a release.
Requires PowerShell 7+.

Modes: `PreCheck` (verify repos are clean), `Collect` (gather entries since last tag),
`Consolidate` (generate release notes), `Full` (all steps).

```powershell
.\Release-Changelog.ps1 -Version "1.0.0"
.\Release-Changelog.ps1 -Mode PreCheck
.\Release-Changelog.ps1 -Version "1.0.0" -SinceTag "v0.9.0" -DryRun
```

### Update-CommitMessage

**Update-CommitMessage.ps1** reads the latest section from `RELEASE_NOTES.md` and writes
it to `.git/MERGE_MSG`, which Visual Studio picks up as the default commit message in
Git Changes. Only processes repositories with uncommitted changes.

```powershell
.\Update-CommitMessage.ps1
.\Update-CommitMessage.ps1 -RepoPath "C:\path\to\PayrollEngine.Backend"
.\Update-CommitMessage.ps1 -RepoPath "C:\path\to\PayrollEngine.Backend" -Silent
```

## Database

Scripts for SQL Server schema export, formatting, diff and version composition.
All scripts handle large `.sql` files (280 KB+) and operate on:

| File | Purpose |
|------|---------|
| `Create-Model.sql` | Full CREATE script for a fresh database |
| `Update-Model.sql` | Incremental migration between two versions |
| `Drop-Model.sql` | DROP script for all schema objects |
| `DbVersion.json` | Version metadata (OldVersion, NewVersion, Descriptions) |

Default location: `C:\...\PayrollEngine.Backend\Database\`

| Script | README | Purpose |
|--------|--------|---------|
| `Export-DbScript.ps1` | [README](Export-DbScript.README.md) | Export the live database schema to a `.sql` file |
| `Format-DbScript.ps1` | [README](Format-DbScript.README.md) | Format a `.sql` file and resolve object dependency order |
| `Compare-DbScript.ps1` | [README](Compare-DbScript.README.md) | Diff two formatted `.sql` files and generate an update script |
| `Generate-DbUpdate.ps1` | [README](Generate-DbUpdate.README.md) | Generate an update script from two SQL files (Format + Compare in one step) |
| `Compose-DbScript.ps1` | [README](Compose-DbScript.README.md) | Full workflow: diff history, replace all regions in-place from `DbVersion.json` |

### Export-DbScript

**Export-DbScript.ps1** connects to SQL Server and exports the live database schema to a
`.sql` file. The connection string is read from the environment variable
`PayrollDatabaseConnection`.

Modes:
- **Create** — generates `CREATE TABLE`, indexes, foreign keys, stored procedures,
  functions and views for the full schema
- **Delete** — generates safe `DROP` statements in dependency order
  (procedures and functions first, then tables)

The output is unordered; pipe it through `Format-DbScript.ps1` to normalise style and
resolve dependency order before committing as a snapshot.

```powershell
# set connection string
$env:PayrollDatabaseConnection = 'Server=.;Database=PayrollEngine;Integrated Security=True;'

.\Export-DbScript.ps1 -Mode Create -TargetFile Snapshot_Create.sql
.\Export-DbScript.ps1 -Mode Delete -TargetFile Snapshot_Delete.sql
```

### Format-DbScript

**Format-DbScript.ps1** formats a `.sql` file and resolves forward-reference dependency
warnings. `SourceFile` is never modified; the result is written to `TargetFile`.

What it does:
- Splits the file into GO-delimited blocks and groups them into named DDL objects
- Detects inter-object dependencies (`EXEC`, UDF calls, `FROM`/`JOIN` references)
- Topologically sorts all objects so each is defined before its dependents
- Eliminates warnings such as:  
  *"The module 'DeleteAllCaseValues' depends on the missing object 'dbo.DeleteAllGlobalCaseValues'..."*
- Normalises GO delimiters, trailing whitespace and line endings (CRLF)

```powershell
.\Format-DbScript.ps1 -SourceFile Create-Model.sql -TargetFile Create-Model.Formatted.sql
.\Format-DbScript.ps1 `
    -SourceFile  C:\...\Database\Create-Model.sql `
    -TargetFile  C:\...\Database\Create-Model.Formatted.sql
```

### Compare-DbScript

**Compare-DbScript.ps1** compares two formatted `.sql` files and generates a delta script
that migrates a database from the baseline state to the current state. Both input files
should be pre-processed by `Format-DbScript.ps1`.

Diff categories:
- **Added** — objects present in Current but not in Baseline → `CREATE`
- **Removed** — objects present in Baseline but not in Current → `DROP`
- **Modified** — objects present in both with different body → `DROP` + `CREATE`  
  (for tables a `TODO` comment is emitted; manual `ALTER TABLE` is required)

```powershell
.\Compare-DbScript.ps1 `
    -BaselineFile Create-Model.Formatted.sql `
    -CurrentFile  Create-Model.New.Formatted.sql `
    -TargetFile   Update-Model.sql
```

### Compose-DbScript

**Compose-DbScript.ps1** orchestrates the full DB script composition workflow in one step.
It diffs `History\v<OldVersion>\Create-Model.sql` against the current `Create-Model.sql`,
then replaces all `#region` blocks in-place in both SQL files:

| Region | File | Content |
|--------|------|---------|
| `DATABASE` | `Create-Model.sql` | Not modified (database setup preamble) |
| `DB_SCRIPTS` | `Create-Model.sql` | Not modified (authoritative schema) |
| `VERSION_SET` | `Create-Model.sql` | Replaced with `NewVersion` insert |
| `VERSION_CHECK` | `Update-Model.sql` | Replaced with `OldVersion` check |
| `DB_SCRIPTS` | `Update-Model.sql` | Replaced with generated delta SQL |
| `VERSION_SET` | `Update-Model.sql` | Replaced with `NewVersion` insert |

See [Compose-DbScript.README.md](Compose-DbScript.README.md) for full documentation.

```powershell
# Run from the Database directory (all defaults)
cd Backend\Database
..\..\devops\scripts\Compose-DbScript.ps1

# Preview without writing
Compose-DbScript.ps1 -DryRun

# Keep intermediate delta file for inspection
Compose-DbScript.ps1 -KeepTemp
```

### Typical Workflow

```powershell
$env:PayrollDatabaseConnection = 'Server=.;Database=PayrollEngine;Integrated Security=True;'
cd Backend\Database

# 1. Compose: diff history + update all regions in-place
..\..\devops\scripts\Compose-DbScript.ps1

# 2. Review Update-Model.sql — replace TODO comments with ALTER TABLE statements

# 3. After release: snapshot into History and advance DbVersion.json
mkdir History\v0.9.6
copy Create-Model.sql  History\v0.9.6\Create-Model.sql
copy Update-Model.sql  History\v0.9.6\Update-Model.sql
copy Drop-Model.sql    History\v0.9.6\Drop-Model.sql
copy DbVersion.json    History\v0.9.6\DbVersion.json
# Update DbVersion.json: OldVersion="0.9.6", NewVersion="0.9.7"
```
